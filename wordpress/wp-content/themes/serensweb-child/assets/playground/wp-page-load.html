How a WordPress page loads, from keystroke to paint

Between typing an address and seeing a page, a request makes a small road trip. Knowing the route is how I make client sites fast: most of the journey can be skipped entirely. Scroll and watch it travel.

Scroll to begin the trip

You hit enter. The browser finds the server.

DNS turns serensweb.com into an IP address, then a TLS handshake sets up the padlock. This usually happens once and gets remembered, but on a first visit it is real time spent before a single byte of the page exists.

typically 20 to 120 ms

The request lands on the web server.

Software like LiteSpeed, Caddy, or Nginx picks up the request. Its first question is not "what does WordPress say?" but "have I answered this exact question recently?"

under 1 ms to route

The page cache check. The fork in the road.

A page cache keeps finished HTML copies of recent pages. If a fresh copy exists, everything below this point on the map simply does not happen. This single yes or no decides most of your load time.

the most important millisecond

Cache hit: the shortcut. Most visitors take this road.

The stored HTML is returned immediately. No PHP runs, no database is touched. On a well-configured site this is the everyday path, and it is why two sites on identical hosting can feel a world apart.

roughly 5 to 30 ms total server time

Cache miss: PHP wakes up and boots WordPress.

WordPress loads its core, the active theme, and every plugin, on every uncached request. This is why I build with zero plugins where possible: each one is code that must wake up before your visitor sees anything.

50 to 400 ms, theme and plugins decide

The database answers (or Redis answers faster).

Content, settings, and menus live in MySQL. WordPress asks dozens of small questions to assemble one page. An object cache like Redis keeps the frequent answers in memory, so repeat questions skip the database queue entirely.

tens of queries, single-digit ms each

The theme renders blocks into HTML.

Templates and block patterns are stitched together with your content into the final document. A lean, well-structured theme renders in a blink; a heavy page builder can spend longer here than everything else combined.

where theme craft shows

The response travels back and gets remembered.

The finished HTML streams to the browser, and a copy usually goes into the page cache so the next visitor takes the shortcut. Compression squeezes it down to a fraction of its size for the trip.

size matters: smaller is faster

Parse and paint. The browser does its half.

HTML is parsed, CSS and fonts load, images decode, and the page paints. Server speed is only half the story: self-hosted fonts, right-sized images, and CSS that does not block are what make this half feel instant.

the half users actually see

What this means for your site

Every fast WordPress site is fast for the same three reasons: most visitors hit a page cache, uncached requests boot the minimum amount of code, and the browser gets lightweight, well-behaved assets. That is the checklist I build against, and it is why this site runs on a custom theme with zero plugins.
